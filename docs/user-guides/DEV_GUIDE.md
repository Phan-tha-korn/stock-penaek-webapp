# คู่มือการใช้งานสำหรับ DEV

บทบาท `DEV` คือผู้ตรวจสอบงานสำคัญ จัดการ flow verification และงานโครงสร้างที่มีผลต่อทั้งระบบ

## 1. เมนูที่ DEV เห็น

### เมนูหลัก
- `งานตรวจสอบของฉัน`
- `การแจ้งเตือน`

### ตรวจสอบงาน
- `คิวตรวจสอบอัตโนมัติ`
- `เทียบสินค้าแบบไว`

### ข้อมูลงาน
- `จัดการสินค้า`
- `จัดการร้านค้า`
- `ธุรกรรม`
- `รายงาน`

### ระบบ
- `จัดการผู้ใช้`
- `ตั้งค่าและหน้าตา`
- `เครื่องมือพัฒนา`

## 2. ลำดับใช้งานที่แนะนำสำหรับ DEV

1. เปิด `งานตรวจสอบของฉัน`
2. เช็ก `คิวตรวจสอบอัตโนมัติ`
3. เปิด `การแจ้งเตือน`
4. ถ้าต้องใช้ข้อมูลประกอบการตัดสินใจ ให้เปิด `เทียบสินค้าแบบไว`
5. ถ้าต้องแก้หรือดูฐานข้อมูลจริง ให้เข้า `จัดการสินค้า`, `จัดการร้านค้า`, หรือ `เครื่องมือพัฒนา`

## 3. เมนูแต่ละตัวใช้ทำอะไร

### 3.1 งานตรวจสอบของฉัน
ใช้เป็นหน้าเริ่มต้นของ DEV

ควรใช้ดู:
- งานที่ assigned ให้ตนเอง
- งานที่เกินเวลา
- งานที่เพิ่งเข้ามา
- งานที่ควรจัดลำดับก่อนหลัง

### 3.2 คิวตรวจสอบอัตโนมัติ
เป็นหน้าหลักที่สุดของ DEV

ใช้ทำอะไร:
- ดูรายการคำขอตรวจสอบทั้งหมด
- ดูงานที่ติดบล็อก
- ดูงานที่เกิน SLA
- หา request แบบเร็ว

วิธีใช้:
1. เปิดหน้า
2. ดู summary ด้านบน
3. ใช้ `ค้นหาแบบไว` ถ้าพิมพ์คำคร่าว ๆ
4. ใช้ filter เฉพาะถ้าจะเจาะ:
   - เลขคำขอ
   - สถานะ
   - ความเสี่ยง
   - ผู้รับผิดชอบ
5. เปิดรายการแล้วตัดสินใจ

สิ่งที่ควรตรวจใน request:
- before / after
- risk level
- dependency warning
- ผู้ขอ
- เหตุผล
- สิ่งที่จะกระทบถ้าอนุมัติ

### 3.3 เทียบสินค้าแบบไว
ใช้ประกอบการตัดสินใจเมื่อ request เกี่ยวข้องกับ:
- ราคา
- ร้านค้า
- กลุ่มเทียบสินค้า
- ต้นทุน

ตัวอย่างการใช้:
- มีการขอเปลี่ยนราคา
- ต้องตรวจว่า supplier ใหม่ราคาดีกว่าจริงไหม
- ต้องดูหลายสินค้าในกลุ่มเดียวกันพร้อมกัน

### 3.4 จัดการสินค้า
ใช้เปิดดูสินค้าต้นทางจริงเมื่อ request อ้างถึงสินค้า

ดูอะไรได้:
- รายละเอียดสินค้า
- รหัสสินค้า
- หมวดหมู่
- ความสัมพันธ์กับกลุ่มเทียบ

### 3.5 จัดการร้านค้า
ใช้ดูร้านค้าที่เกี่ยวข้องกับ request

ดูอะไรได้:
- รายละเอียดร้าน
- คะแนนความน่าเชื่อถือ
- สถานะ verify
- ไฟล์แนบ
- ประวัติ proposal

### 3.6 ธุรกรรม
ใช้ดู movement ที่เกี่ยวกับงานนั้น ๆ หากต้องตรวจผลกระทบทางข้อมูล

### 3.7 รายงาน
ใช้ดูมุมมองย้อนหลังและแนวโน้ม

### 3.8 การแจ้งเตือน
ใช้เป็น inbox ของ DEV

ควรเปิดดูเมื่อ:
- มีงานใหม่
- มีงานเกินเวลา
- มีงานที่โดน escalate
- มี pricing change สำคัญ

### 3.9 จัดการผู้ใช้
ใช้เมื่อจำเป็นต้องตรวจ role / scope / assignment

### 3.10 ตั้งค่าและหน้าตา
ใช้ปรับการใช้งานส่วนตัว เช่น ภาษา โหมดสี และหน้าแสดงผล

### 3.11 เครื่องมือพัฒนา
ใช้เฉพาะงานเทคนิค เช่น:
- เช็กข้อมูล debug
- รีเซ็ตบางข้อมูลตาม workflow dev
- ตรวจสภาพระบบในเชิงพัฒนา

## 4. งานหลักที่ DEV ต้องทำให้ครบในแต่ละ request

1. เปิด request
2. อ่านเหตุผล
3. ดู before / after
4. ดู risk / dependency
5. เปิดข้อมูลจริงประกอบถ้าจำเป็น
6. ถ้าเกี่ยวกับราคา ให้เปิด `เทียบสินค้าแบบไว`
7. ตัดสินใจ:
   - approve
   - reject
   - return for revision
   - escalate
8. เช็กว่าอนุมัติแล้วมี:
   - snapshot
   - audit
   - notification

## 5. ตัวอย่าง flow ที่ DEV ใช้จริงบ่อย

### Supplier Change
1. notification เด้ง
2. เปิดคิวตรวจสอบ
3. ดู proposal + attachment
4. เปิดหน้าร้านค้าจริง
5. approve หรือ reject

### Pricing Change
1. เปิด request
2. เปิดเทียบสินค้าแบบไว
3. เช็กราคาปัจจุบันกับราคาที่จะเปลี่ยน
4. ดูผลกระทบ
5. approve

### Matching Change
1. ดูว่างานเป็น move / merge / split
2. เช็ก dependency
3. ดูสินค้าที่เกี่ยวข้อง
4. approve หรือ escalate

## 6. สิ่งที่ DEV ต้องระวัง

- อย่าอนุมัติงานที่ blocked โดยไม่เข้าใจ dependency
- ถ้าเทียบราคาแล้วจำนวนไม่ตรง ต้องเช็ก quantity ที่ใช้เทียบก่อน
- งานที่เกี่ยวกับหลายสินค้า ควรใช้ compare แบบหลายรายการ ไม่ดูตัวเดียว
- ถ้าข้อมูลไม่พอ ให้ return for revision ดีกว่า approve แบบเดา
